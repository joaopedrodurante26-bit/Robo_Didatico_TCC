Eu proporia que o sistema de testes deixasse de ser apenas um "modo de testes" e se tornasse uma **interface de manutenção, diagnóstico e validação do robô**, funcionando como um terminal de serviço integrado ao firmware. A ideia central é que qualquer componente do robô possa ser testado individualmente, sem a necessidade de modificar o código-fonte, comentar trechos do programa ou criar sketches temporários para cada hardware. Dessa forma, o console passa a ser uma ferramenta permanente do projeto, utilizada durante o desenvolvimento, na depuração, na manutenção e até mesmo na demonstração do funcionamento do robô durante a apresentação do TCC.

O console funcionará através da porta serial do ESP32. Assim que o firmware for inicializado, o sistema executará normalmente toda a sequência de boot, inicializando cada módulo do robô (logger, motores, sensores, Wi-Fi, sistema de arquivos, controle etc.). Durante essa inicialização, serão exibidas mensagens indicando o progresso e o resultado de cada etapa, permitindo ao desenvolvedor identificar imediatamente eventuais falhas de hardware ou de software.

Após o término da inicialização, o sistema apresentará um cabeçalho informando que o robô está pronto para receber comandos. Em seguida, será exibido um prompt semelhante ao de um terminal de computador, indicando que o console está aguardando a entrada do usuário.

Um exemplo da inicialização seria:

```text
=========================================================
 ROBÔ EDUCACIONAL
 Console de Diagnóstico e Manutenção
 Firmware 1.0
=========================================================

Inicializando sistema...

[ OK ] Logger
[ OK ] Motores
[ OK ] Controle
[ OK ] Encoders
[FAIL] MPU6050
[ OK ] WiFi
[FAIL] LittleFS

Sistema inicializado.

Digite HELP para listar os comandos.

ROBO>
```

O prompt representa o estado atual do console e indica ao usuário que novos comandos podem ser digitados.

## Arquitetura baseada em estados

O console não será implementado como uma sequência de comandos independentes. Em vez disso, será organizado como uma máquina de estados.

Inicialmente, o console permanece no **modo principal**, responsável apenas pelos comandos gerais do sistema. Quando o usuário desejar testar um determinado subsistema, será possível entrar em um submenu específico. Cada submenu possuirá seus próprios comandos, exibirá um prompt próprio e interpretará apenas as instruções relacionadas ao componente correspondente.

Por exemplo, ao digitar:

```text
ROBO> MOTOR
```

o console mudará internamente seu estado para "Modo Motores". A partir desse momento, o prompt será alterado para:

```text
MOTOR>
```

Isso informa ao usuário que todos os comandos enviados serão interpretados pelo módulo responsável pelos motores.

Da mesma forma poderão existir estados específicos para sensores, Wi-Fi, sistema de arquivos, diagnóstico e outros módulos.

Essa abordagem evita que todos os comandos do projeto precisem ser analisados simultaneamente, tornando a implementação mais organizada e reduzindo a possibilidade de conflitos entre comandos.

# Organização dos menus

O console será dividido em módulos, cada um responsável por uma área do sistema.

## Sistema

Responsável pelo gerenciamento geral do firmware.

Exemplo de comandos:

```text
HELP
STATUS
VERSION
REBOOT
CLEAR
```

Esses comandos permitem consultar informações gerais, limpar o terminal, reiniciar o ESP32 ou exibir ajuda.

---

## Motores

O menu de motores permitirá controlar diretamente cada motor do robô, ignorando completamente os algoritmos de navegação, controle PID ou qualquer outra camada de abstração.

Seu objetivo é validar exclusivamente o hardware responsável pela locomoção.

Ao entrar nesse menu:

```text
ROBO> MOTOR
```

o console responderá:

```text
====================================
 TESTE DOS MOTORES
====================================

Comandos disponíveis

F <velocidade>

T <velocidade>

VE <velocidade>

VD <velocidade>

E <velocidade>

D <velocidade>

STOP

EXIT

MOTOR>
```

Os comandos terão o seguinte significado:

* **F** movimenta os dois lados para frente.
* **T** movimenta os dois lados para trás.
* **VE** realiza uma rotação para a esquerda.
* **VD** realiza uma rotação para a direita.
* **E** controla apenas o lado esquerdo.
* **D** controla apenas o lado direito.
* **STOP** interrompe imediatamente todos os motores.
* **EXIT** retorna ao menu principal.

Todos esses comandos utilizarão diretamente as funções públicas do módulo `motores`, garantindo que exatamente o mesmo código utilizado durante a operação normal do robô seja testado.

# Sensores

O submenu de sensores será dividido em diversas categorias.

### MPU6050

Permitirá visualizar as leituras do acelerômetro e do giroscópio.

Exemplo:

```text
SENSOR> MPU READ
```

Resposta:

```text
Accel

X = 0.01 g

Y = -0.03 g

Z = 0.98 g

Gyro

X = 0.2 °/s

Y = -0.1 °/s

Z = 0.0 °/s
```

Também poderá existir um modo contínuo:

```text
MPU STREAM
```

Nesse modo, os valores serão atualizados continuamente até que o usuário digite:

```text
STOP
```

---

### Encoders

Permitirá verificar individualmente a contagem de pulsos de cada roda.

Exemplo:

```text
ENCODER READ
```

Resposta:

```text
Encoder esquerdo

Pulsos = 154

Encoder direito

Pulsos = 151
```

Também poderão existir comandos para zerar os contadores ou acompanhar a leitura continuamente.

---

### Ultrassônico

Permitirá medir continuamente a distância detectada.

```text
ULTRA READ
```

Resposta:

```text
Distância

53.4 cm
```

Ou:

```text
ULTRA STREAM
```

para atualização contínua.

Exemplo:


Resultado:

```text

 DIAGNÓSTICO AUTOMÁTICO
==============================
```

Esse recurso permite verificar rapidamente se todo o hardware está operacional antes de iniciar uma aula ou uma demonstração.

# Estrutura interna do console

Embora o usuário enxergue apenas um terminal simples, internamente o sistema será completamente modular.

O console ficará responsável apenas por:

* receber caracteres da porta serial;
* montar a linha digitada pelo usuário;
* identificar o estado atual do console;
* encaminhar o comando para o módulo correspondente.

Cada módulo do projeto será responsável exclusivamente por interpretar seus próprios comandos.

Essa separação evita dependências desnecessárias entre os componentes e mantém a arquitetura coerente com o restante do firmware.

---

# Objetivos do sistema

A criação desse console atende a diversos objetivos técnicos:

* permitir a validação individual de cada módulo de hardware;
* reduzir significativamente o tempo gasto durante a depuração do sistema;
* evitar a necessidade de recompilar o firmware para cada teste específico;
* reutilizar as mesmas funções empregadas durante o funcionamento normal do robô, aumentando a confiabilidade dos testes;
* facilitar a identificação de falhas elétricas, problemas de comunicação ou erros de configuração;
* servir como ferramenta permanente de manutenção durante todo o ciclo de vida do projeto.

Além de apoiar o desenvolvimento, esse console representa uma importante característica de engenharia de software embarcado. Em vez de um conjunto de testes temporários, ele constitui uma infraestrutura de diagnóstico integrada ao firmware, permitindo que o robô seja inspecionado, testado e mantido de forma sistemática. Essa abordagem demonstra preocupação com modularidade, testabilidade e manutenção, aspectos frequentemente valorizados em projetos de engenharia e que agregam qualidade técnica ao TCC.
